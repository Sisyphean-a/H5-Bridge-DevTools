# 桥接发送、响应与匹配分层管理

## 用户故事

- 作为经常调试桥接的前端或测试，我希望把发送桥接和响应桥接分开查看，而不是在一条规则里反复确认哪些字段属于发送、哪些字段属于响应。
- 作为需要在同一个发送桥接下切换不同结果的人，我希望能在多个响应候选之间快速切换当前生效项，而不必复制和修改整条规则。
- 作为排查单向桥接的人，我希望能明确看到某个发送桥接当前是“无自动响应”，而不是把没有回调误判成异常。
- 作为验证页面监听逻辑的人，我希望每个响应桥接都可以单独一键触发，便于只验证响应消费链路。

## 为什么需要

规则数量一多，发送条件、响应内容、当前配对关系混在同一编辑面板里会迅速失去可读性。尤其是同一发送桥接要切换成功、失败等不同结果时，用户很难快速确认当前生效的是哪条响应，也不容易复用已有响应配置。

## 怎么解决

规则页作为总入口，但内部按“发送”“响应”“匹配”拆成三个子页签。发送页专注管理匹配条件，响应页专注管理响应内容并支持单独触发，匹配页专注查看和切换当前配对关系。一个发送桥接可以挂多个响应候选，但同一时刻只允许一个活跃响应；关闭配对后，该发送进入“无自动响应”状态。

## 边界

- 这次改造只重构规则模型和规则页交互，不改变桥接拦截、日志记录和页面回放的基础机制。
- 当前关联仍围绕桥接事件名工作，不按 `requestId`、`callbackId` 等实例级字段做精确关联。
- 一个发送桥接可以拥有多个响应候选，但同一时刻只允许一个活跃响应，不支持一次发送同时触发多个自动响应。
- 响应页中的“一键触发”用于单独验证响应效果，不等价于重新发起一次原生发送桥接。
