# Iteration 001

## Current Understanding

现有存储已经按 origin 共享，缺的是已打开页面和面板的自动消费链路；响应编辑器已有 JSON 文本能力，缺的是图片文件到常见桥接字段的便捷写回。

## Implementation Approach

- 复用 `chrome.storage.local` 作为共享状态源，在内容脚本层订阅 `chrome.storage.onChanged`，收到共享 key 变化后重载运行态并重新向面板发布快照。
- 保留现有 JSON 编辑器，新增图片工具层，把文件转换结果写回 `responseDraft.detailText`，不改响应数据模型。

## Changes This Iteration

- 在内容脚本中接入 `chrome.storage.onChanged` 监听，并在存储变化后重载同 origin 快照再发布 `SNAPSHOT`。
- 在运行态中提取 `setRuntimeSnapshot` / `reloadRuntimeSnapshot`，区分初始化时是否继承历史日志与实时同步时必须接收共享日志。
- 为响应编辑器增加 `ResponseImageTools`，支持图片选择、预览、目标字段填写、模拟 Android URI / Base64 切换，并写回 `detail` JSON。
- 新增 `imageTools` 纯函数与测试，覆盖字段推荐、点路径写回和模拟 URI 生成。
- 新增运行态测试，覆盖初始化与实时同步对日志处理的差异。

## Verification Evidence

- `npm run typecheck`：通过
- `npm test`：通过，7 个测试文件 / 31 个测试全部通过
- `npm run build`：通过
- 独立功能验收：图片工具通过；跨页同步因缺少真实多页面现场证据，结论为 `inconclusive`

## Problems Encountered

- 扩展的跨页同步很难只靠单元测试证明“现场无缝”，因此独立验收没有把这部分直接判通过。
- 面板原本就有脏草稿保留策略；它能防止本地未保存编辑被覆盖，但也会让远端更新的显示存在滞后风险。

## Next Attempt

在真实扩展环境做同 origin 多页面 / 多 DevTools 现场验收，决定是否需要为脏草稿场景增加远端更新提示。
