# H5 Bridge DevTools Chrome Extension 项目需求文档

## 1. 项目名称

**H5 Bridge DevTools**

也可以命名为：

* Native Bridge Mock DevTools
* H5 Native Bridge Mock
* Bridge Lab DevTools

推荐最终名称：**H5 Bridge DevTools**。

---

## 2. 项目背景

我是一个 H5 前端开发者，日常开发的 H5 页面运行在 Android 宿主 App 内。很多核心业务流程必须依赖 Native Bridge，例如：

* 登录
* 获取用户信息
* 打开相机
* 选择联系人
* 活体检测
* 获取定位
* 打开外部浏览器
* 跳转订单详情
* Native 代理请求
* 上传大 JSON
* 上报 Airbridge / 埋点

问题是：这些桥接能力在普通电脑浏览器里不存在，导致 H5 页面在 Chrome 浏览器里无法完整跑通业务流程。开发时经常必须依赖 Android 同事、测试包、真机宿主环境，效率很低。

我希望做一个 Chrome DevTools 扩展，在 F12 里新增一个自定义 Tab，用来模拟 Android Native Bridge。它需要能够：

1. 在页面启动前注入 `window.AndroidBridge`。
2. 捕获 H5 调 Native 的 bridge 请求。
3. 根据规则自动 mock Native 回调。
4. 支持 Native 主动向 H5 推送消息。
5. 支持规则新增、修改、删除、启用、禁用、导入、导出。
6. 支持手动发送 Native 消息，单独测试 H5 监听逻辑。
7. 支持桥接调用日志，方便调试参数和返回结果。

这个工具的目标不是完全模拟 Android WebView，而是让我在电脑 Chrome 浏览器里能跑通 80% 的 H5 主流程。

---

## 3. 产品定位

这是一个 **Chrome Manifest V3 DevTools 扩展**。

它不是普通 popup 插件，而是一个类似 Vue Devtools、Redux DevTools 的 F12 面板。

打开 Chrome DevTools 后，顶部工具栏里应该出现一个新的 Tab：

```txt
Elements | Console | Sources | Network | Application | H5 Bridge
```

点击 **H5 Bridge** 后进入桥接 mock 面板。

---

## 4. 核心目标

### 4.1 第一目标

让本地 Chrome 浏览器具备类似 Android 宿主的桥接环境：

```js
window.AndroidBridge.postMessage(...)
```

页面代码不需要修改，也能在浏览器中调用桥接。

---

### 4.2 第二目标

支持自动回调。

例如页面调用：

```js
window.AndroidBridge.postMessage(JSON.stringify({
  event: "openCamera"
}))
```

扩展捕获到 `openCamera` 后，根据规则在 1000ms 后自动执行：

```js
window.dispatchEvent(new CustomEvent("openCamera", {
  detail: {
    success: true,
    uri: "mock://camera/photo-001.jpg",
    data: "mock-image-data"
  }
}))
```

这样 H5 原有的 `waitForAndroidMessage("openCamera")` 就能收到结果。

---

### 4.3 第三目标

所有规则都通过 UI 表单简单配置，不要求用户写复杂代码。

规则数据使用 JSON 存储和导入导出，但日常新增修改尽量通过表单完成。

---

## 5. 功能边界

### 5.1 要做的事情

第一版需要实现：

1. Chrome Manifest V3 扩展。
2. DevTools 自定义 Panel：`H5 Bridge`。
3. `document_start` 阶段注入 `window.AndroidBridge`。
4. 支持 `window.AndroidBridge.postMessage(message)`。
5. 自动解析 `message`，如果是字符串则尝试 `JSON.parse`。
6. 捕获 bridge 请求并展示日志。
7. 根据 `event` 字段匹配 mock 规则。
8. 按规则延迟发送 Native 回调。
9. 支持 `dispatchEvent` 回调模式。
10. 支持手动 Emit Native 消息。
11. 支持规则 CRUD：

    * 新增
    * 修改
    * 删除
    * 启用
    * 禁用
    * 复制
12. 支持导入 / 导出规则 JSON。
13. 支持内置模板：

    * 登录成功
    * 登录失败
    * 相机成功
    * 相机取消
    * 联系人成功
    * 联系人失败
    * 活体成功
    * 活体失败
    * 定位成功
    * 上传大 JSON 成功
    * baseRequest 成功
14. 支持从日志创建规则。
15. 支持复制调用参数和复制 mock 响应。
16. 支持按当前页面 origin 保存规则。
17. 支持全局 Mock 开关。

---

### 5.2 暂时不做的事情

第一版不要做太大，以下功能暂时不做：

1. 不真实调用 Android 相机。
2. 不真实读取通讯录。
3. 不真实模拟 Android 权限弹窗。
4. 不真实模拟 WebView 生命周期。
5. 不真实模拟 Android 返回键。
6. 不真实模拟前后台切换。
7. 不做远程 JS 规则执行。
8. 不做复杂脚本引擎。
9. 不做云端账号体系。
10. 不做团队协作平台。
11. 不做复杂 HTTP mock，HTTP mock 可以交给 MSW、Whistle、Reqable 等工具。

---

### 5.3 可以后续扩展的事情

第二阶段可以考虑：

1. 支持 iOS bridge：


3. 支持 callbackName 回调模式。

5. 支持远程 JSON 规则同步。
6. 支持规则分组和场景切换。
7. 支持按项目 / 包名 / appId 区分规则。
8. 支持导出日志。
9. 支持 mock 图片文件预览。
10. 支持把日志发送到本地服务。

## 6. 设计理念

### 6.1 工具应该偏工程化，不要偏运营后台

这个工具是开发者每天用的，不是给运营人员看的后台系统。

UI 要像开发工具，而不是像传统管理后台。

关键词：

* 简洁
* 高信息密度
* 极客风
* 暗色优先
* 数据清晰
* 操作直接
* 少弹窗
* 少大卡片
* 少花哨渐变
* 少无意义图标
* 少营销感

---

### 6.2 视觉风格偏好

我喜欢的风格：

* 简约
* 偏极客
* 像 Logcat / Network / Console 工具
* 信息密度高
* 层级清晰
* 黑灰底色
* 细边框
* 小字号
* 等宽字体用于 JSON / 日志
* 状态颜色清楚但克制

不喜欢的风格：

* 大面积圆角卡片
* 过度留白
* 运营后台风
* 花哨渐变
* 复杂装饰
* 很多没用的插画
* 像 SaaS 官网一样的 UI
* 一堆大按钮、大标题、大卡片

### 8.4 右侧 Logs 面板

日志显示桥接调用和 mock 回调。

日志类型：

* `SEND`：H5 调用 Native。
* `MOCK`：扩展自动模拟 Native 回调。
* `EMIT`：用户手动发送 Native 消息。
* `WARN`：没有匹配规则。
* `ERROR`：JSON 解析失败或发送失败。

日志示例：

```txt
10:23:11.123 SEND openCamera
{
  "event": "openCamera"
}

10:23:12.125 MOCK openCamera
{
  "success": true,
  "uri": "mock://camera/photo-001.jpg"
}
```

每条日志支持：

* 展开 / 收起
* 复制 payload
* 复制 response
* 从这次调用创建规则
* 重新发送 mock response
* 仅显示此 event
* 清空日志

### 8.5 Manual Emit 面板

中间区域或底部提供一个手动发送区。

用途：即使页面没有主动调用 bridge，也可以手动模拟 Native 主动通知 H5。

字段：

```txt
Event Name:
openCamera

Detail JSON:
{
  "success": true,
  "uri": "mock://camera/photo-001.jpg"
}

[Send to Page]
```

点击后在页面执行：

```js
window.dispatchEvent(new CustomEvent(eventName, {
  detail
}))
```

并记录一条 `EMIT` 日志。

## 9. 当前 H5 Bridge 协议

第一版主要适配 Android bridge。

### 9.1 H5 调 Native

H5 会调用：

```js
window.AndroidBridge.postMessage(serializedMessage)
```

其中 `serializedMessage` 通常是 JSON 字符串。

常见格式：

```json
{
  "event": "openCamera"
}
```

或者：

```json
{
  "event": "getContact"
}
```

或者：

```json
{
  "event": "startLiveness",
  "h5Json": {
    "type": 1,
    "accountId": "xxx",
    "key": "xxx"
  }
}
```

或者 baseRequest：

```json
{
  "event": "baseRequest",
  "requestType": "1",
  "requestUrl": "/api/xxx",
  "requestHeader": {},
  "requestParams": {}
}
```

---

### 9.2 Native 回 H5

第一版采用 `dispatchEvent` 模式。

扩展模拟 Native 时执行：

```js
window.dispatchEvent(new CustomEvent(eventName, {
  detail: detailJson
}))
```

H5 的 receiver 会把它解析成：

```ts
{
  event: event.type,
  dataJson: event.detail
}
```

因此：

```js
window.dispatchEvent(new CustomEvent("openCamera", {
  detail: {
    success: true,
    uri: "mock://camera/photo.jpg"
  }
}))
```

会被 H5 识别为：

```ts
{
  event: "openCamera",
  dataJson: {
    success: true,
    uri: "mock://camera/photo.jpg"
  }
}
```

## 11. 规则数据结构

第一版规则接口：

```ts
export interface BridgeMockRule {
  id: string
  name: string
  enabled: boolean

  match: {
    event: string
  }

  response: {
    delayMs: number
    mode: "dispatchEvent"
    eventName: string
    detail: unknown
  }

  meta?: {
    createdAt?: number
    updatedAt?: number
    hitCount?: number
  }
}
```

## 12. 日志数据结构

```ts
export type BridgeLogType = "SEND" | "MOCK" | "EMIT" | "WARN" | "ERROR"

export interface BridgeLogItem {
  id: string
  type: BridgeLogType
  event?: string
  timestamp: number
  payload?: unknown
  response?: unknown
  message?: string
  ruleId?: string
}
```

## 13. 存储设计

使用 `chrome.storage.local`。

按 origin 存储规则：

```ts
export interface BridgeStorageState {
  globalEnabled: boolean
  origins: Record<string, OriginBridgeState>
}

export interface OriginBridgeState {
  rules: BridgeMockRule[]
  logs?: BridgeLogItem[]
  settings?: {
    autoMock: boolean
    preserveLogs: boolean
    maxLogCount: number
  }
}
```

key 示例：

```txt
h5BridgeDevTools:v1
```

默认配置：

```ts
{
  globalEnabled: true,
  origins: {
    "http://localhost:5173": {
      rules: presetRules,
      settings: {
        autoMock: true,
        preserveLogs: false,
        maxLogCount: 200
      }
    }
  }
}
```

## 15. 通信架构

### 15.1 整体链路

```txt
H5 页面
  ↓
window.AndroidBridge.postMessage(...)
  ↓
injectMain.ts，运行在 MAIN world，页面可访问
  ↓ window.postMessage
contentScript.ts，运行在 ISOLATED world，可访问 chrome.storage
  ↓ chrome.runtime.sendMessage / port
background service worker
  ↓ port
DevTools Panel
```

---

### 15.2 为什么需要 MAIN world

普通 content script 默认运行在隔离环境，页面 JS 无法访问它里面挂载的变量。

因此注入 `window.AndroidBridge` 必须运行在 MAIN world。

---

### 15.3 为什么还需要 content script

MAIN world 脚本适合挂载 `window.AndroidBridge` 和执行页面回调，但不适合直接做扩展存储、DevTools 通信等逻辑。

因此：

* `injectMain.ts`：只做页面环境能力。
* `contentScript.ts`：负责规则执行、日志采集、存储读取、消息转发。
* `panel.tsx`：负责 UI。
* `serviceWorker.ts`：负责 DevTools panel 和 tab/content script 的消息桥。

---

## 16. injectMain.ts 职责

`injectMain.ts` 运行在页面 MAIN world。

职责：

1. 在 `document_start` 注入。
2. 如果页面已经存在 `window.AndroidBridge`，不要强行覆盖，除非 Mock 开关允许覆盖。
3. 提供：

   ```js
   window.AndroidBridge.postMessage(message)
   ```
4. 收到 postMessage 后，转发给 content script：

   ```js
   window.postMessage({
     source: "H5_BRIDGE_DEVTOOLS_PAGE",
     type: "BRIDGE_CALL",
     payload: {
       rawMessage,
       parsedMessage
     }
   }, "*")
   ```
5. 监听 content script 发来的 Native 回调命令：

   ```js
   window.addEventListener("message", ...)
   ```
6. 执行：

   ```js
   window.dispatchEvent(new CustomEvent(eventName, { detail }))
   ```

---

## 17. contentScript.ts 职责

`contentScript.ts` 运行在 isolated world。

职责：

1. 监听页面发来的 `BRIDGE_CALL`。
2. 解析 bridge message。
3. 记录 SEND 日志。
4. 读取当前 origin 的规则。
5. 找到匹配 event 的第一条 enabled rule。
6. 如果找到规则：

   * 增加 hitCount。
   * 按 delayMs 延迟。
   * 发送命令给 MAIN world 执行回调。
   * 记录 MOCK 日志。
7. 如果没有找到：

   * 记录 WARN 日志。
   * DevTools 面板显示未匹配。
8. 接收 DevTools Panel 的手动 Emit 命令。
9. 将日志同步给 Panel。

## 19. 核心实现逻辑

### 19.1 注入 AndroidBridge

```ts
(function installAndroidBridgeMock() {
  const SOURCE_PAGE = "H5_BRIDGE_DEVTOOLS_PAGE"
  const SOURCE_EXTENSION = "H5_BRIDGE_DEVTOOLS_EXTENSION"

  if (window.AndroidBridge?.postMessage) {
    window.__H5_BRIDGE_ORIGINAL_ANDROID_BRIDGE__ = window.AndroidBridge
  }

  window.AndroidBridge = {
    postMessage(message) {
      let parsedMessage = null

      try {
        parsedMessage = typeof message === "string" ? JSON.parse(message) : message
      } catch {
        parsedMessage = null
      }

      window.postMessage({
        source: SOURCE_PAGE,
        type: "BRIDGE_CALL",
        payload: {
          rawMessage: message,
          parsedMessage
        }
      }, "*")
    }
  }

  window.addEventListener("message", event => {
    const data = event.data

    if (!data || data.source !== SOURCE_EXTENSION) return

    if (data.type === "DISPATCH_EVENT") {
      window.dispatchEvent(new CustomEvent(data.payload.eventName, {
        detail: data.payload.detail
      }))
    }
  })
})()
```

---

### 19.2 规则匹配

第一版只按 `event` 匹配：

```ts
function findRule(rules: BridgeMockRule[], event: string) {
  return rules.find(rule => rule.enabled && rule.match.event === event)
}
```

后续可以扩展：

* 按 `h5Json` 匹配
* 按 `requestUrl` 匹配
* 按正则匹配
* 按条件组合匹配

---

### 19.3 自动 Mock 回调

```ts
async function handleBridgeCall(call: BridgeCallPayload) {
  const event = call.parsedMessage?.event

  addLog({
    type: "SEND",
    event,
    payload: call.parsedMessage
  })

  if (!event) {
    addLog({
      type: "ERROR",
      message: "Bridge message has no event field.",
      payload: call.rawMessage
    })
    return
  }

  const rule = findRule(currentRules, event)

  if (!rule) {
    addLog({
      type: "WARN",
      event,
      message: `No mock rule matched for event "${event}".`,
      payload: call.parsedMessage
    })
    return
  }

  setTimeout(() => {
    postToMainWorld({
      type: "DISPATCH_EVENT",
      payload: {
        eventName: rule.response.eventName,
        detail: rule.response.detail
      }
    })

    addLog({
      type: "MOCK",
      event: rule.response.eventName,
      response: rule.response.detail,
      ruleId: rule.id
    })
  }, rule.response.delayMs)
}
```

## 23. 安全原则

### 23.1 不执行远程 JS

Manifest V3 不允许扩展执行远程下发的 JS。这个工具必须遵守：

允许：

```json
{
  "eventName": "openCamera",
  "detail": {
    "success": true
  }
}
```

不允许：

```json
{
  "script": "window.someFunction()"
}
```

规则只能是 JSON 数据，由扩展内置逻辑解释执行。

---

### 23.2 不加载不可信远程脚本

扩展所有 JS 逻辑都必须打包在扩展内。

---

### 23.3 不默认对所有网站开启

第一版建议只匹配：

* localhost
* 127.0.0.1
* 指定测试域名

后续可以允许用户手动添加域名。

---

### 23.4 避免污染页面

虽然需要注入 `window.AndroidBridge`，但应尽量减少其他全局变量。

允许添加：

```js
window.AndroidBridge
window.__H5_BRIDGE_ORIGINAL_ANDROID_BRIDGE__
```

不应该大量挂载无关变量。
